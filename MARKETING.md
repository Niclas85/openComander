# Marketing-Plan fuer OpenCommander

OpenCommander ist ein kostenloser, quelloffener Android-Dateimanager fuer Nutzer, die lokal und ohne Cloud-Zwang mit Dateien arbeiten wollen.

## Positionierung

**Ein lokaler Zwei-Seiten-Dateimanager fuer Android: kostenlos, Open Source, ohne Werbung, ohne Tracking, ohne Konto.**

Zielgruppe:

- Android-Nutzer, die Dateien aktiv organisieren, kopieren, verschieben und archivieren.
- Nutzer, die klassische Zwei-Fenster-Dateimanager moegen.
- Datenschutzbewusste Nutzer, die keine Cloud-Pflicht, Werbung oder Analyse-SDKs wollen.
- Power-User, Techniker, Admins, Entwickler und Nutzer mit vielen lokalen Dateien.

Kernversprechen:

- Zwei Seiten, zwei Ordnerbaeume, schnelles Kopieren und Verschieben.
- ZIP-Dateien wie Ordner durchsuchen und eigene ZIPs erstellen.
- Rueckgaengig-Historie fuer versehentliche Dateiaktionen.
- Lokal, transparent, kostenlos und Open Source.

## App-Store-Optimierung

Empfohlener Titel:

`OpenCommander - File Manager`

Alternativen:

- `OpenCommander File Explorer`
- `OpenCommander - Dual Pane Files`
- `OpenCommander - Local File Manager`

Kurzbeschreibung:

`Kostenloser Open-Source-Dateimanager mit zwei Fenstern, ZIP, Drag & Drop und Rueckgaengig.`

Keyword-Schwerpunkte:

- file manager
- file explorer
- dateimanager
- zwei fenster
- dual pane
- commander
- zip manager
- local files
- open source
- no ads
- privacy friendly

Nicht verwenden:

- Fremde Marken als App-Titel oder Hauptclaim.
- Uebertriebene Versprechen wie "sicherster Dateimanager".
- Claims, die Cloud- oder Backup-Funktionen suggerieren.

## Screenshot-Reihenfolge

1. **Zwei Seiten. Schneller arbeiten.**
   - Querformat mit beiden Seiten zeigen.

2. **Kopieren oder verschieben per Drag & Drop.**
   - Datei markiert, Zielseite sichtbar.

3. **Rueckgaengig mit Historie.**
   - Historie offen, Rueckgaengig sichtbar.

4. **ZIPs wie Ordner oeffnen.**
   - ZIP im Baum oder in Dateiliste sichtbar.

5. **Kostenlos. Open Source. Keine Werbung.**
   - Ruhige Ansicht mit Datenschutz-/AGB-Hinweis oder sauberer Dateiliste.

## Community-Kanaele

P0:

- GitHub-Repository mit README, Screenshots, Lizenz und Releases: https://github.com/Niclas85/openComander
- F-Droid-Einreichung pruefen, sobald Release-Signing und Metadaten stabil sind.
- Reddit/Foren nur mit ehrlichem Build-Log posten, nicht werblich spammen.

P1:

- Kleine Demo-Clips: "copy files in two panes", "undo a move", "open ZIP like folder".

- GitHub Issues mit Labels: `bug`, `feature`, `good first issue`, `translation`.

## 7-Tage-Plan

1. Store-Listing aus `playstore/listing-de.md` und `playstore/listing-en.md` finalisieren.
2. Lizenz und Anbieterangaben pruefen.
3. README mit Screenshots und Open-Source-Hinweis auf GitHub veroeffentlichen: https://github.com/Niclas85/openComander
4. Feature Graphic und Screenshots mit den empfohlenen Claims abgleichen.
5. Privacy/Terms auf einer oeffentlichen URL hosten.
6. Testrelease intern verteilen.
7. Feedback zu Navigation, Icons, ZIP und Rueckgaengig einsammeln.

## 30-Tage-Plan

- Google Play Internal Testing oder Closed Testing.
- GitHub Releases mit APK/AAB-Hinweisen.
- F-Droid-Metadaten vorbereiten.
- 3 kurze Demo-Videos erstellen.
- Erste Community-Rueckmeldungen in Issues ueberfuehren.

## 90-Tage-Plan

- Stabile 1.0-Version.
- F-Droid-Submission.
- Mehr Sprachen und bessere Store-Lokalisierung.
- Optional: eigene kleine Website mit Datenschutz, Screenshots, Downloadlinks und GitHub-Link.

## Messung ohne Tracking

Da die App bewusst ohne Analytics auskommt, sollten externe, datensparsame Signale genutzt werden:

- Store-Installationen und Deinstallationsrate.
- GitHub Stars, Issues, Releases-Downloads.
- F-Droid-Downloadzahlen, falls verfuegbar.
- Qualitatives Feedback aus Reviews und Issues.
- Manuelle Testliste fuer Aktivierung: erste Datei markieren, kopieren, ZIP oeffnen, Rueckgaengig nutzen.
